# AGENTS.md

## Правила общения

- Всегда отвечать на русском языке, если пользователь явно не попросил другой язык.
- Писать простым человеческим языком, без лишнего профессионального жаргона.
- Если тема сложная, сначала коротко объяснить смысл, потом детали.
- Если пользователь прислал скриншот, учитывать его как важный контекст.
- Если пользователь спрашивает про настройку, код, ошибку или новую тему, сначала объяснить смысл простыми словами.

## Подача ответа

- Не писать рваными короткими строками в столбик.
- По умолчанию отвечать связными абзацами, а не россыпью отдельных строк.
- Оптимальный формат: короткое объяснение, затем 1-2 компактных абзаца с деталями.
- Списки использовать только когда они реально помогают: для шагов, вариантов или чек-листа.
- Если есть несколько вариантов, сначала сказать, какой рекомендуется и почему.
- Не использовать длинное тире. Вместо него использовать короткое тире: `-`.
- Стараться писать коротко, понятно и доступно, избегая сложных английских терминов.

## Работа с кодом

- Перед заметными изменениями коротко объяснять, что будет сделано.
- Не удалять и не перезаписывать файлы пользователя без явной причины.
- Если есть риск сломать проект, сначала предупредить.
- Если есть уже существующие изменения, не откатывать их без прямой просьбы пользователя.
- После изменений запускать релевантную проверку, если понятно, какую именно.

## Правила для этого проекта

- Основные источники информации: `ТЗ для тг бота.docx` и `ответы на вопросы для бота.docx`.
- Если промпт и Word-файлы расходятся, приоритет у Word-файлов.
- Первая версия проекта - MVP Telegram-бота, без полноценной админ-панели.
- Архитектуру делать простой и расширяемой, чтобы позже можно было добавить админ-панель.
- Не выдумывать недостающие данные. Если информации нет, ставить понятную заглушку и добавлять пункт в список уточнений.
- Базу знаний хранить отдельно от кода, чтобы ответы можно было редактировать без изменения логики бота.
- Токен бота, ID администраторов и изменяемые настройки хранить вне кода, через `.env` и конфиги.
- Навигацию делать через Inline-кнопки.
- После каждого ответа должны быть кнопки `Назад` и `Главное меню`.
- Пользователь должен иметь возможность вернуться назад или в главное меню из любого раздела.

## Ведение плана

- Основной план хранится в `plan.md`.
- Каждый новый план должен быть с текущей датой.
- План разбивается на фазы, фазы - на пункты.
- Каждый пункт должен иметь чекбокс.
- После выполнения пункта нужно менять `[ ]` на `[x]`.
